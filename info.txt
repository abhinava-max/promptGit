PromptGitX Workflow and Function Reference
==========================================

Project name:
PromptGitX

Purpose:
PromptGitX is a CLI tool that reviews Git changes using an LLM. It can analyze
commits, multiple commits, branch comparisons, pull requests, the latest commit,
the last N commits, or staged changes. It prints a terminal report and can save
the report as txt, json, docx, or pdf.


1. Main CLI Commands
====================

Entry command:
promptgitx

Main commands:

1. promptgitx
   Shows the app header and welcome screen.

2. promptgitx --version
   Prints the installed PromptGitX version and exits.

3. promptgitx chat
   Starts the AI chat interface placeholder for Git commit messages.
   Current behavior: prints "Chat is running".

4. promptgitx config
   Configures the LLM provider, model list, API key, or local base URL.

5. promptgitx analyze
   Generates an AI review report for a Git target.


2. Config Command
=================

Command:
promptgitx config [OPTIONS]

Options:

--provider, -p
  Selects the LLM provider.
  Accepted values:
  - groq
  - openai
  - anthropic
  - gemini
  - ollama

--models, -m
  Comma-separated model list.
  Maximum 5 models.
  Example:
  promptgitx config --provider openai --models gpt-4.1,gpt-4.1-mini --api-key YOUR_KEY

--api-key
  API key for cloud providers.
  Used for Groq, OpenAI, Anthropic, and Gemini.

--base-url
  Base URL for local providers.
  Mainly used for Ollama.
  Default interactive value:
  http://localhost:11434

--reset, -r
  Deletes the local .env configuration file.

Config workflow:
1. User runs promptgitx config.
2. Provider is selected from CLI options or interactive menu.
3. For cloud providers, an API key is collected.
4. For Ollama, a base URL is collected.
5. Model names are parsed and limited to 5.
6. Values are saved into .env.
7. CURRENT_PROVIDER is updated.

.env keys written:
CURRENT_PROVIDER
GROQ_API_KEY
OPENAI_API_KEY
ANTHROPIC_API_KEY
GEMINI_API_KEY
OLLAMA_BASE_URL
<PROVIDER>_MODEL_1
<PROVIDER>_MODEL_2
<PROVIDER>_MODEL_3
<PROVIDER>_MODEL_4
<PROVIDER>_MODEL_5


3. Analyze Command
==================

Command:
promptgitx analyze [TARGET OPTION] [OUTPUT OPTION]

Only one target option can be used at a time.

Target options:

--commit, -c <commit_hash>
  Reviews one specific commit.
  Internally uses:
  git show --format= --unified=80 <commit_hash>

--commits, -C <commit_hash>
  Reviews multiple commits.
  This option can be provided multiple times.
  Example:
  promptgitx analyze -C abc123 -C def456
  Internally fetches each commit diff and joins them.

--compare, -p <left..right>
  Reviews the diff between two refs.
  Examples:
  promptgitx analyze --compare main..feature-login
  promptgitx analyze --compare HEAD~3..HEAD
  Internally uses:
  git diff <left>..<right> --unified=80

--pr, -P <number>
  Reviews a GitHub pull request.
  Requires GitHub CLI.
  Internally uses:
  gh pr diff <number>

--last, -l
  Reviews the latest commit.
  Internally uses:
  git show --format= --unified=80 HEAD

--last-n, -n <number>
  Reviews the combined diff from the last N commits.
  Internally uses:
  git diff HEAD~N..HEAD --unified=80

--staged, -s
  Reviews staged changes.
  Internally uses:
  git diff --cached --unified=80

Output options:

--json
  Prints the raw structured JSON report.

--summary
  Prints only the short summary report.

--save <path>
  Saves the report to a file.
  Supported extensions:
  - .txt
  - .json
  - .docx
  - .pdf

Examples:

promptgitx analyze --last
  Reviews the latest commit and prints a rich terminal report.

promptgitx analyze --staged --summary
  Reviews staged changes and prints only the summary.

promptgitx analyze --commit abc123 --json
  Reviews commit abc123 and prints raw JSON.

promptgitx analyze --compare main..feature --save report.pdf
  Reviews the branch comparison and saves a PDF.

promptgitx analyze --last --save report.docx
  Reviews the latest commit and saves a DOCX.


4. Analyze Workflow
===================

High-level flow:

1. CLI receives command.
2. main.py validates selected options.
3. generate_report() starts the review workflow.
4. LangGraph loads the requested Git diff.
5. The diff is parsed into file-level chunks.
6. Each file chunk is sent to the configured LLM.
7. LLM output is parsed as JSON.
8. File reviews are normalized.
9. A final structured report is built.
10. The report is printed in terminal, summary, or JSON form.
11. If requested, the report is saved as txt, json, docx, or pdf.

LangGraph node order:

START
  -> load_diff
  -> parse_diff
  -> review_chunks
  -> final_report
  -> END


5. Report Data Structure
========================

The final report dictionary contains:

target:
  The selected review target.
  Example:
  {"mode": "commit", "value": "abc123"}

overall_summary:
  A short natural-language summary.

summary:
  files_reviewed
  lines_added
  lines_removed
  risk_level
  final_recommendation

files:
  List of reviewed files.
  Each file contains:
  file_path
  summary
  added_lines
  removed_lines
  issues

issues:
  Grouped report sections:
  critical
  security
  breaking_changes
  vulgarity
  performance
  code_quality
  improvements

Allowed issue categories:
bug
breaking_change
security
performance
code_quality
vulgarity
standards
testing
documentation
maintainability

Allowed severities:
low
medium
high
critical

Final recommendation values:
APPROVE
NEEDS_MANUAL_REVIEW
REQUEST_CHANGES

12. Full End-to-End Example
===========================

Example:
promptgitx analyze --staged --save review.pdf

What happens:
1. Typer calls analyze().
2. get_analyze_mode() returns "staged".
3. generate_report() receives mode="staged".
4. run_review_graph() starts the LangGraph workflow.
5. load_diff_node() calls get_staged_diff().
6. get_staged_diff() runs git diff --cached --unified=80.
7. parse_diff_node() calls summarize_diff() and chunk_diff_by_file().
8. review_chunks_node() sends each file diff to the configured LLM.
9. extract_json_object() parses the LLM response.
10. normalize_file_review() cleans each file review.
11. final_report_node() calls build_report().
12. generate_report() prints the Rich terminal report.
13. save_report() sees the .pdf extension.
14. create_pdf_report() builds and writes review.pdf.

14. Exact LangGraph Architecture
================================

LangGraph file:
src/promptgitx/ai/review_graph.py

State type:
ReviewGraphState

State definition file:
src/promptgitx/ai/graph_state.py

LangGraph imports used:
StateGraph
START
END

Graph creation function:
create_review_graph()

Graph execution function:
run_review_graph(initial_state)

Graph shape:

START
  -> load_diff
  -> parse_diff
  -> review_chunks
  -> final_report
  -> END

There are no conditional branches in the current graph.
The workflow is linear and deterministic.
The selected analyze mode changes what load_diff_node() does, but the graph path
itself stays the same for every review.


15. What LangGraph Does In This Project
=======================================

LangGraph is used as the orchestration layer for the review pipeline.

Instead of writing one long function that does everything, the review is split
into named nodes. Each node receives a shared state dictionary and returns new
state fields. LangGraph merges those returned fields into the running state and
passes the updated state to the next node.

In this project, LangGraph does not make the AI smarter by itself.
Its job is to organize the workflow:

1. Load the correct Git diff.
2. Parse and chunk the diff.
3. Review each chunk with the LLM.
4. Build the final report.

This makes the architecture easier to extend later. For example, future versions
could add conditional edges for:
- skipping LLM review when there is no diff
- routing security issues to a separate security-review node
- retrying failed model responses
- reviewing large files with a different strategy
- adding a final human-readable summarizer node


16. ReviewGraphState Fields
===========================

ReviewGraphState is a TypedDict with total=False.
That means every field is optional from Python's type-checking perspective.
Nodes add fields gradually as the workflow progresses.

Input fields passed into run_review_graph():

mode
  The selected analyze mode.
  Example values:
  commit
  commits
  compare
  pr
  last
  last_n
  staged

commit
  Commit hash for --commit.

commits
  List of commit hashes for --commits.

compare
  Compare expression for --compare.
  Example:
  main..feature

pr
  Pull request number for --pr.

last
  Boolean flag for --last.

last_n
  Number of commits for --last-n.

staged
  Boolean flag for --staged.

Fields added by load_diff_node():

diff
  Raw Git diff text fetched from git or gh.

repo_context
  Repository metadata used as prompt context.
  Currently includes repository root and current branch when available.

model_name
  Display name of the currently configured model.

Fields added by parse_diff_node():

diff_summary
  Dictionary containing total files, added lines, removed lines, and per-file
  line counts.

chunks
  List of file-level diff chunks.
  Each chunk contains:
  file_path
  added_lines_count
  removed_lines_count
  changed_lines
  raw_diff
  was_trimmed

Fields added by review_chunks_node():

chunk_reviews
  List of raw per-chunk review records.
  Each item stores chunk number, file path, and normalized review.

file_reviews
  List of normalized file reviews used by report_builder.py.

Fields added by final_report_node():

report
  Final structured report dictionary.

final_report
  Pretty JSON string version of the final report.


17. LangGraph Node Details
==========================

Node 1:
load_diff

Function:
load_diff_node(state)

Purpose:
Loads the correct Git diff based on state["mode"].

Mode routing inside this node:

commit
  Calls get_commit_diff(state.get("commit")).

commits
  Calls get_multiple_commits_diff(state.get("commits") or []).

compare
  Calls get_compare_diff(state.get("compare")).

pr
  Calls get_pr_diff(state.get("pr") or 0).

last
  Calls get_last_commit_diff().

last_n
  Calls get_last_n_commits_diff(state.get("last_n") or 0).

staged
  Calls get_staged_diff().

Returns:
diff
repo_context
model_name


Node 2:
parse_diff

Function:
parse_diff_node(state)

Purpose:
Turns raw diff text into summary data and file-level chunks.

Calls:
summarize_diff(diff)
chunk_diff_by_file(diff)
filter_large_diff_chunks(chunks, max_chars=MAX_CHUNK_CHARS)

Important constant:
MAX_CHUNK_CHARS = 12000

This means very large file diffs are trimmed before being sent to the model.

Returns:
diff_summary
chunks


Node 3:
review_chunks

Function:
review_chunks_node(state)

Purpose:
Reviews each file-level diff chunk with the configured LLM.

Chain used:
get_chunk_review_prompt()
  | create_chat_model_with_fallbacks()
  | StrOutputParser()

This is LangChain expression language.
It means:
1. Build the prompt.
2. Send it to the configured chat model.
3. Convert the model output into a string.

For each chunk, the node sends:
mode
repo_context
file_path
added_lines_count
removed_lines_count
numbered_changed_lines
raw_diff

numbered_changed_lines:
This is generated by format_numbered_changed_lines(chunk).
It gives the model exact line references computed from Git hunk headers.
Example:
src/app.py:L42: + return user.name

The model is instructed to use only those exact references.

After model response:
1. extract_json_object(raw_review) parses JSON.
2. normalize_file_review(parsed_review, chunk) validates and normalizes data.
3. normalize_line_reference() checks whether the model's line_reference is real.

If model parsing fails:
The node creates a fallback medium maintainability issue that says the AI review
could not be parsed.

Returns:
chunk_reviews
file_reviews


Node 4:
final_report

Function:
final_report_node(state)

Purpose:
Builds the final structured report.

Calls:
build_report(state=state, file_reviews=state.get("file_reviews", []))
to_pretty_json(report)

Returns:
report
final_report

19. Why LangGraph Is Useful Here
================================

LangGraph gives the review process clear boundaries.

Benefits in this project:

1. Each step has one responsibility.
   load_diff_node fetches data.
   parse_diff_node prepares data.
   review_chunks_node calls the LLM.
   final_report_node builds output.

2. The state is explicit.
   Each node adds named fields, making debugging easier.

3. The graph can be extended without rewriting the whole analyzer.
   New nodes can be inserted between existing nodes.

4. Error handling can become more advanced later.
   For example, a failed model response could go to a retry node.

5. It matches AI-agent workflows well.
   AI systems often need multiple steps: gather context, reason, validate,
   produce output. LangGraph gives structure to that process.


20. Current LangGraph Limitations In This Project
=================================================

Current limitations:

1. The graph is linear.
   There are no conditional edges yet.

2. Each file chunk is reviewed sequentially.
   There is no parallel chunk review yet.

3. There is no retry node.
   Invalid model JSON becomes a fallback issue immediately.

4. There is no separate final LLM summary node.
   The final report is currently built in Python by report_builder.py.

5. There is no persistent checkpointing.
   If a long review stops halfway, it starts over next time.

Possible future improvements:

1. Add conditional edge:
   If no chunks exist, skip review_chunks and go directly to final_report.

2. Add retry node:
   If JSON parsing fails, ask the model to repair its JSON once.

3. Add security specialist node:
   If security-related lines are detected, run a deeper security review.

4. Add parallel chunk review:
   Review multiple files concurrently to reduce runtime.

5. Add checkpointing:
   Save progress between nodes for large reviews.

6. Add final summarizer node:
   Let the LLM write a polished executive summary after Python builds the
   structured report.
